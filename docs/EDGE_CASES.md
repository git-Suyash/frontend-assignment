# Edge Cases

| Edge Case | How Triggered | Detection Mechanism | User-Facing Behavior | State Outcome |
|---|---|---|---|---|
| **Price tampering** | `window.__devTools.simulatePriceTampering(id, newPrice)` in browser console â€” mutates `product.price` in localStorage while leaving `snapshotPrice` unchanged | `detectPriceTampering()` in `priceValidation.ts` compares `item.product.price !== item.snapshotPrice` for every cart item at checkout time | Error toast "Tampering detected". Checkout blocked. Idempotency key reset. | `CART_READY` (validation fails, no transition) |
| **Checksum mismatch** | `window.__devTools.simulateChecksumTampering()` sets `cart.checksum = 'TAMPERED_HASH'` in localStorage then reload | `computeCartChecksum(items)` is recomputed at checkout and compared to stored `cartState.checksum` | Error toast "Security check failed". Checkout blocked. | `CART_READY` |
| **IDB baseline integrity failure** | Open DevTools â†’ Application â†’ IndexedDB â†’ `opencart_catalog` â†’ `price_baseline` â†’ edit the `integrity` field to an arbitrary hex string | `verifyBaselineIntegrity()` recomputes SHA-256 of canonical `id:price` entries; mismatch triggers `BASELINE_INTEGRITY_FAILED` audit event | Error toast "Tampering detected". Checkout blocked. | `CART_READY` |
| **IDB baseline price mismatch** | Edit a product's `price` field inside the `price_baseline` IDB record to match a tampered `snapshotPrice` in localStorage (while also forging the FNV-1a checksum) | `validateAgainstBaseline()` compares each `snapshotPrice` against the corresponding entry in the IDB baseline; a delta `> 0.01` triggers `BASELINE_TAMPERING_DETECTED` | Error toast "Tampering detected". Checkout blocked. | `CART_READY` |
| **IDB baseline missing** | Clear IndexedDB (Application â†’ Storage â†’ Clear site data) then attempt checkout without reloading the page | `getPriceBaseline()` returns `null`; `BASELINE_MISSING` audit event is logged; checkout proceeds with the remaining 9 checks still active | No UI block â€” checkout continues. `BASELINE_MISSING` written to audit log. | Checkout continues |
| **Stale prices (API drift)** | Product price changes on server between add-to-cart and checkout | `detectStalePrice()` in `priceValidation.ts`: re-fetches all products via `fetchProducts()` and checks `|snapshotPrice - freshPrice| > 0.01` | Warning toast "Prices changed". Checkout blocked, cart remains intact for review. | `CART_READY` |
| **Cross-tab cart conflict** | Open two tabs, add different items in each | `window.addEventListener('storage', ...)` in `CartProvider` (`src/contexts/CartProvider.tsx`) compares incoming `parsed.version > state.version` | Amber conflict banner in CartPage. Warning toast "Cart conflict". | `status: 'conflict'` on CartState |
| **Idempotency re-use** | Rapid double-submit or replay of a previously consumed checkout key | `consumeIdempotencyKey()` checks localStorage list of used keys (last 10). Returns `false` if key was already consumed. | Warning toast "Duplicate attempt". Checkout blocked. New key generated on next attempt. | `CART_READY` |
| **Order inconsistent (partial failure)** | `simulatePartialFailure()` returns `true` (15% probability) â€” API succeeded but local persistence "failed" | Simulated in `orders.ts`: after a successful 201 response, throws `PARTIAL_FAILURE` to simulate a split-brain scenario | Error toast "Order status unclear". User is shown retry/rollback UI. Recovery banner shows on refresh. | `ORDER_INCONSISTENT` |
| **Refresh during ORDER_SUBMITTED** | User closes/refreshes browser tab while order is processing | `OrderContext` persists state on every change. On mount, `loadOrder()` restores non-terminal state. `CheckoutPage` detects `ORDER_SUBMITTED` on mount and shows recovery UI | Recovery UI: "Interrupted checkout â€” Resume or Start over" | `ORDER_SUBMITTED` (restored from localStorage) |
| **Empty cart checkout attempt** | User navigates directly to `/checkout` with an empty cart | Check 1 in `validateCheckout()`: `items.length === 0` | Warning toast "Empty cart". Redirect to `/`. | `CART_READY` (unchanged) |
| **Cart locked double-submit** | Multiple rapid clicks on "Place Order" while submission is in progress | Check 2 in `validateCheckout()`: `cartState.status === 'locked'`; also button disabled via `disabled={submitting}` | UI button disabled. If somehow triggered, validation returns `CART_LOCKED`. | `CART_READY` |
